# Architecture

## The one rule

Dependencies point **inward only**:

```
data/source (Tanzil, immutable)
      │  npm run build
      ▼
src/build  ──writes──▶  data/generated/quran-analytics.json   (the contract)
                              │  read once at startup
                              ▼
                        src/domain   (pure functions, no I/O after load)
                              │
                              ▼
                        src/http     (thin transport; any other transport can replace it)
```

- `src/domain` never imports from `src/build` or `src/http`.
- `src/http` contains **no domain logic** — every handler is a one-line call into `src/domain`.
- The dataset JSON is the only coupling between build-time and run-time. Any other runtime
  (Python, a CDN, a spreadsheet) can consume the same artifact.

## Layers

### `src/build` — the pipeline (runs offline, `npm run build`)

| Module | Responsibility |
|---|---|
| `parse-tanzil-sql.mjs` | Tanzil SQL dump → 6236 `{sura, aya, text}` rows; fails on any irregularity |
| `parse-tanzil-metadata.mjs` | Tanzil metadata JS → surahs, juz/rub/manzil/ruku/page boundaries, sajdas; strips 1-indexing placeholders and end sentinels; fails on unexpected shape |
| `count-text.mjs` | The counting rules (pure). Every character must be classified counted/ignored or the build **throws** — nothing is dropped silently |
| `generate-dataset.mjs` | Orchestrates, cross-validates both sources against each other, runs hand-audited spot checks, aggregates divisions, serializes |

Validation is not a separate phase that can be skipped — the generator refuses to write a
dataset that fails any invariant (row counts, per-surah lengths, juz↔rub alignment, division
sums vs whole-Quran totals, letter/word spot checks).

### `data/generated/quran-analytics.json` — the contract

Versioned via `meta.version`. Serialized deterministically (one row per line) so diffs are
reviewable. Committed to git: consumers never need to run the pipeline.
Field-by-field reference: [DATA_DICTIONARY.md](DATA_DICTIONARY.md).

### `src/domain` — pure queries

| Module | Responsibility |
|---|---|
| `dataset.js` | Load JSON once; build indexes (maps, flat arrays, prefix sums, division start indexes); `globalIndex`/`refOfGlobal` |
| `structure.js` | Surah/division/sajda lookups; `locate(surah, ayah)` → full mushaf position |
| `stats.js` | Letter/word/ayah counts and percentages at every granularity |
| `selectors.js` | The selector grammar (`"2:1-141"`, `"juz:30"`, …) → ayah membership mask |
| `progress.js` | Letter-weighted (or word-/ayah-weighted) scope-relative progress |

Design choices worth knowing:

- **Global ayah index (1..6236)** is the universal coordinate. Every division is a contiguous
  range of it, so range sums are O(1) via prefix arrays and "which juz is 2:255 in" is a binary
  search.
- **Selector masks** are a `Uint8Array(6237)`. Composing/deduplicating arbitrary selections
  becomes trivial bitwise work — no interval-merging logic to get wrong.
- Lookups return `null`/`0` for out-of-range inputs; selectors **throw** `SelectorError`
  (user input deserves a message; programmatic lookups deserve a cheap miss).

### `src/http` — transport

`router.js` is a pure function `(method, URL) → {status, body}` — unit-tested without sockets.
`server.mjs` wraps it in `node:http` with CORS + caching headers. Zero dependencies, so the
whole API deploys anywhere Node 18+ runs.

**Porting to another platform:** keep `src/domain`, replace `src/http`:
- Express/Fastify: call `handleRequest` from one middleware.
- Serverless/Workers: bundle domain + JSON; map the platform request to a `URL`.
- Static hosting: publish per-endpoint JSON files generated by walking the router.

## Extension points

- **New division type** (e.g. Warsh numbering, thumn): add boundaries in the build, an entry in
  `DIVISION_TYPES` — selectors, stats, locate and HTTP routes pick it up automatically.
- **New weight** (e.g. Uthmani glyph width, audio seconds): add a per-ayah array in the build
  and one entry in `WEIGHTS`.
- **Ayah text / translations:** deliberately excluded (stats-only keeps the artifact small and
  license handling simple). If added, put them in a separate generated file, not this one.

## Testing strategy

- `tests/dataset.test.js` — the artifact itself (shape, totals, audited spot values, division sums).
- Domain tests compare against **independent reimplementations** (e.g. `divisionAt` vs a linear
  scan), not against the code under test.
- `tests/http.test.js` — router unit tests plus one real socket round-trip.
